# 变更管理（Change Management, ECN / DCN）

## ① 角色定位（给谁用，具体岗位画像）

本技能服务于组织内部"变更"的正式受控管理岗位：

- **变更工程师 / 项目经理**：变更的发起与归口人，负责组织影响分析、串联评审、推动闭环。
- **设计 / 工艺工程师（PE）**：判定 ECN 技术可行性与验证准则；DCN 中负责文件编制。
- **质量工程师（QE）**：评估变更对质量/合规的影响，参与会签与验证确认。
- **文控（DCC）**：DCN 的审核、受控发放、旧版回收与分发管理。

画像要点：岗位处于"一个变更要不要做、怎么做、谁批准、怎么切换"的决策链上，需要在变更生效前建立一道"受控闸门"，避免变更漏评、文件不同步、未验证先切换。

## ② 使用场景（工作环节 / 触发条件 / 手头材料）

**工作环节**：产品/文件生命周期中的"变更提出 → 影响分析 → 评审批准 → 验证 → 切换 → 闭环"环节。

**触发条件**（满足任一即启用）：
- 客户要求变更（功能/性能/规格）。
- 设计优化、降本、可制造性改进。
- 纠正措施（8D）触发的设计/工艺变更。
- 体系审核不符项、法规/CSR 更新导致的文件或产品变更。
- 供应商变更需经本企业 ECN 评审后采用。

**手头材料**（用户提供，缺项标注「待补充」）：
- 变更类型（ECN 工程变更 / DCN 文件变更）。
- 变更对象（零件号/图号/版本，或文件名称/编号/版本）。
- 变更内容与原因。
- 影响分析要素（4M、成本、交期、法规、关联产品/文件）。
- 验证方案与评审/批准结论。
- 切换计划与生效时点。

## ③ 解决的问题（业务痛点，非动作）

- **漏评关联影响**：只改主文件，漏同步 FMEA/控制计划/检验规范，现场执行与文件"两张皮"。
- **未验证先切换**：变了但没做首件/首批/型式验证，问题在客户端才暴露。
- **编号混乱不可追溯**：变更无编号/无版本，发生不良时无法定位是否由变更引起。
- **审批越级/缺会签**：责任悬空，复盘互相推诿。
- **供应商变更失控**：供应商改了本企业不知道、未走 ECN 评审直接采用。

## ④ 能力边界（能做 / 不做 / 输入要求）

**能做**：
- 按 ECN（改产品技术状态）或 DCN（只改文件）分类，生成受控变更记录（纯文字版 TXT + Markdown）。
- 引导做影响分析（4M/成本/交期/法规/关联文件清单）与评审路由、验证方案、切换计划。
- 内置 ECN/DCN 小样本演示，即时产出记录。

**不做**：
- 不替代 PLM/ERP/文控系统：本技能只产出受控记录文档，企业系统流程优先。
- 不替用户定放行/批准结论：会签结论与批准权限由相关人判定。
- 不编造零件号、图号、版本号、客户名；缺项标注「待补充」。
- 不预设批准矩阵：谁有批准权依企业授权，技能只提示"需跨职能会签/文控发放"。

**输入要求**：至少提供"变更类型（ECN/DCN）+ 变更对象 + 变更内容"。未提供类型时先引导判别（改产品=ECN，改文件=DCN），不生成空记录。

## ⑤ 交互与角色设定（语气 / 采集方式 / 是否强制交互锁 / 引导深度）

- **语气**：严谨、受控、强调"闸门"意识；像变更评审员，常用"影响分析、会签、验证、切换、闭环"。
- **采集方式**：按类型逐项引导；用户一次性给全则直接生成记录。
- **强制交互锁**：**是**。未确认"变更类型"与"评审/批准结论（或待批准）"前，不标记为闭环；缺对象不生成空表。缺项标注「待补充」。
- **引导深度**：中等。引导到变更"可分析、可评审、可追溯、可切换"即可；深究技术机理时提示转相邻技能（FMEA/PPAP/控制计划）。

## ⑥ 处理流程

1. **判别类型**：改产品技术状态 → ECN；只改文件 → DCN。
2. **采集对象**：零件号/图号/版本（ECN）或文件名称/编号/版本（DCN）。
3. **影响分析**：4M、成本、交期、法规/CSR、关联产品；ECN 必查关联文件清单（BOM/FMEA/CP/SOP/检验规范/图纸）。
4. **评审路由**：ECN 跨职能会签（设计/工艺/质量/采购/生产）+ 批准；DCN 编制→审核→文控发放。
5. **验证方案**：首件/首批/型式试验/PPAP；未验证 = 禁止切换放行。
6. **切换计划**：生效时点、旧版处置、新旧断点（批次/序列号）。
7. **生成记录**：调用 `scripts/build_report.py` 产出纯文字版 TXT + Markdown。

## ⑦ 输入输出（格式 + 示例）

**输入**：交互采集或用户提供的变更清单（每条含类型、对象、内容、影响分析、评审、验证、切换等）。

**输出**：受控变更记录（纯文字版 + Markdown）：
- `ECN变更记录_<编号>.txt` / `.md`（工程变更）
- `DCN变更记录_<编号>.txt` / `.md`（文件变更）

产物写到用户当前工作目录（可 `--out-dir` 指定）。

**示例（ECN 核心字段，节选）**：

| 项目 | 内容 |
|------|------|
| 编号 | ECN-2026-0713-01 |
| 变更对象 | P-10023 出水阀密封圈，版本 C→D |
| 变更类型 | 材料 |
| 影响分析 | 料/法变更；单件成本 +0.3 元；满足客户耐温 CSR |
| 关联文件 | BOM/DFMEA/控制计划/检验规范 同步更新 |
| 验证 | 首批 200 件型式试验 + PPAP 等级3 |
| 批准 | 设计/工艺/质量会签，周经理批准 |

> ECN 改产品技术状态；DCN 只改文件不改实物。

## ⑧ 内置知识 / 参考清单

- **ECN/DCN 判别**：改产品=ECN，改文件=DCN（详见 `references/change-mgmt-knowledge.md`）。
- **影响分析维度**：4M、成本、交期、法规/CSR、关联产品；ECN 关联文件清单（BOM/FMEA/CP/SOP/检验规范/图纸）。
- **评审路由**：ECN 跨职能会签+批准；DCN 编制→审核→文控(DCC)发放。
- **标准条款索引**：ISO 9001:2015 8.3.6（设计更改）/ 8.5.6（生产更改）；IATF 16949 变更管理。企业 PLM/文控流程优先。
- **常见坑**：漏同步关联文件、未验证先切换、编号混乱、审批越级、供应商变更未走 ECN。

## ⑨ 联动声明（相邻技能，纯提示）

- 变更已在现场发生 → 变化点管理 `change-point-management`（N11，执行侧受控记录）。
- 供应商入向变更 → 参考技能 `供应商变更管理`（常需先过本企业 ECN 评审）。
- 变更触发 FMEA/PPAP/控制计划更新 → 对应 FMEA / PPAP / 控制计划技能。
- 变更作为纠正措施 → `8d-problem-solving`。
- 开发期设计变更 → APQP 阶段管理。
